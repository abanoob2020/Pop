# دراسة بحثية: نظام إدارة الجيم (Gym Management System)

> وثيقة بحث وتخطيط أولية لمشروع "Pop" — نظام متكامل لإدارة صالات الألعاب الرياضية.
> التاريخ: 2026-07-03

---

## 1. نظرة عامة على المشروع

نظام إدارة الجيم هو تطبيق يهدف إلى أتمتة العمليات اليومية لصالة الألعاب الرياضية:
إدارة الأعضاء والاشتراكات، تسجيل الحضور، إدارة المدربين والحصص التدريبية،
المدفوعات والفواتير، والتقارير الإدارية.

### الأهداف الرئيسية
1. تقليل العمل اليدوي والورقي في تسجيل الأعضاء ومتابعة الاشتراكات.
2. منع تسرّب الإيرادات (اشتراكات منتهية، مدفوعات متأخرة).
3. تحسين تجربة العضو (حجز الحصص، متابعة التقدم، تنبيهات التجديد).
4. توفير رؤية إدارية عبر تقارير ولوحات معلومات.

---

## 2. أصحاب المصلحة والأدوار (Roles)

| الدور | الوصف | أهم الصلاحيات |
|-------|-------|----------------|
| **مدير النظام (Admin)** | مالك الجيم أو المدير العام | كل الصلاحيات، التقارير المالية، إدارة الموظفين |
| **موظف الاستقبال (Receptionist)** | يتعامل مع الأعضاء يوميًا | تسجيل أعضاء، تحصيل مدفوعات، تسجيل حضور |
| **المدرب (Trainer)** | يدير الحصص والبرامج التدريبية | جدول الحصص، متابعة أعضائه، خطط التمرين |
| **العضو (Member)** | المشترك في الجيم | عرض اشتراكه، حجز حصص، متابعة تقدمه |

## 3. الميزات الوظيفية (Functional Requirements)

### 3.1 إدارة الأعضاء
- تسجيل عضو جديد (بيانات شخصية، صورة، جهة اتصال للطوارئ، بيانات صحية أساسية).
- بحث وتصفية الأعضاء (بالاسم، رقم الهاتف، حالة الاشتراك).
- ملف العضو: سجل الاشتراكات، المدفوعات، الحضور، القياسات الجسدية.

### 3.2 الاشتراكات والباقات
- تعريف باقات مرنة: شهري، ربع سنوي، سنوي، حصص محدودة (مثلاً 12 حصة).
- تجميد الاشتراك (Freeze) لمدة محددة.
- تجديد وترقية الباقة مع احتساب الفرق.
- تنبيهات آلية قبل انتهاء الاشتراك (رسالة/إشعار).

### 3.3 الحضور (Check-in)
- تسجيل دخول العضو عند البوابة (رقم عضوية، QR code، أو بصمة لاحقًا).
- رفض الدخول تلقائيًا للاشتراكات المنتهية مع تنبيه موظف الاستقبال.
- إحصاءات أوقات الذروة.

### 3.4 المدفوعات والفواتير
- تسجيل المدفوعات (نقدي، بطاقة، تحويل) مع إيصال قابل للطباعة.
- دعم الدفع الجزئي والمديونيات.
- سجل مالي يومي/شهري (إغلاق الوردية).

### 3.5 الحصص والمدربون
- جدول أسبوعي للحصص الجماعية (سعة قصوى لكل حصة).
- حجز/إلغاء حجز الحصص من طرف العضو.
- إسناد أعضاء لمدرب خاص (PT) وجلسات خاصة.

### 3.6 التقارير ولوحة المعلومات
- الإيرادات (يومي/شهري/سنوي) حسب الباقة وطريقة الدفع.
- الأعضاء النشطون، الجدد، المنتهية اشتراكاتهم، معدل الاحتفاظ (Retention).
- نسب إشغال الحصص وأداء المدربين.

### 3.7 ميزات مستقبلية (خارج النطاق الأولي)
- تطبيق موبايل للأعضاء.
- بوابات دفع إلكتروني (Stripe / Paymob / Tap حسب السوق).
- إدارة مخزون (مكملات، مشروبات) ونقطة بيع (POS).
- تكامل أجهزة بصمة/بوابات دخول.

## 4. المتطلبات غير الوظيفية

- **اللغة:** واجهة ثنائية اللغة (عربي RTL + إنجليزي) من البداية.
- **الأمان:** مصادقة JWT/جلسات، صلاحيات حسب الدور (RBAC)، تشفير كلمات المرور (bcrypt/argon2).
- **الخصوصية:** البيانات الصحية للأعضاء حساسة — أقل قدر ممكن من الجمع، وتقييد الوصول.
- **العمل دون اتصال (اختياري):** الاستقبال قد يحتاج تحمّل انقطاع الإنترنت إن كان النظام سحابيًا.
- **قابلية التوسع:** دعم أكثر من فرع (Multi-branch) كتصميم مستقبلي في نموذج البيانات.

```
Member        (id, name, phone, email, gender, birth_date, photo, emergency_contact, notes, created_at)
Plan          (id, name, duration_days, sessions_limit, price, is_active)
Subscription  (id, member_id, plan_id, start_date, end_date, sessions_used, status[active|frozen|expired|cancelled], price_paid)

Trainer       (id, name, phone, specialty)
GymClass      (id, name, trainer_id, weekday, start_time, duration_min, capacity)
ClassBooking  (id, class_id, member_id, date, status[booked|attended|cancelled])
User          (id, name, username, password_hash, role[admin|receptionist|trainer])
Measurement   (id, member_id, date, weight, height, body_fat, notes)
```

### العلاقات الأساسية
- عضو ← اشتراكات متعددة (تاريخي)، الاشتراك النشط واحد فقط في أي لحظة.
- الاشتراك ← مدفوعات متعددة (دفع جزئي).
- الحصة ← حجوزات بحد أقصى = السعة.

## 6. الخيارات التقنية (Tech Stack)

بما أن المستودع فارغ ولا توجد قيود تقنية سابقة، هذه أبرز الخيارات:

### الخيار A — تطبيق ويب كامل بـ Next.js + PostgreSQL ⭐ (الموصى به)
- **Frontend + Backend:** Next.js (App Router) + TypeScript.
- **قاعدة البيانات:** PostgreSQL مع Prisma ORM.
- **الواجهة:** Tailwind CSS + مكتبة مكونات تدعم RTL (مثل shadcn/ui).
- **المميزات:** مكدس واحد، تطوير سريع، مجتمع ضخم، نشر سهل.
- **العيوب:** يتطلب اتصالًا بالإنترنت إن استُضيف سحابيًا.

### الخيار B — API منفصل (Node/NestJS أو Django) + واجهة React
- فصل أوضح بين الطبقات، مناسب إذا خُطط لتطبيق موبايل قريبًا (نفس الـ API).
- تكلفة تطوير أولية أعلى (مشروعان).

### الخيار C — تطبيق سطح مكتب محلي (Electron/Tauri + SQLite)
- مناسب لجيم واحد بدون إنترنت موثوق.
- يصعب التوسع لعدة فروع أو وصول الأعضاء من الخارج.

**التوصية:** البدء بالخيار A مع بنية نظيفة تسمح باستخراج API لاحقًا لتطبيق الموبايل.

## 7. خارطة الطريق المقترحة (Roadmap)

### المرحلة 1 — الأساس (MVP)
1. إعداد المشروع (Next.js + TypeScript + Prisma + PostgreSQL) والمصادقة والأدوار.
2. إدارة الأعضاء (CRUD + بحث).
3. الباقات والاشتراكات + التجديد.
4. تسجيل المدفوعات مع إيصالات.
5. تسجيل الحضور مع التحقق من صلاحية الاشتراك.

### المرحلة 2 — التشغيل اليومي
6. لوحة معلومات وتقارير أساسية (إيرادات، اشتراكات منتهية قريبًا).
7. جدول الحصص والحجوزات.
8. تنبيهات انتهاء الاشتراك.

### المرحلة 3 — التوسع
9. القياسات الجسدية وخطط التدريب.
10. بوابة/تطبيق العضو.
11. الدفع الإلكتروني وتعدد الفروع.

## 8. المخاطر وأسئلة مفتوحة

| السؤال | الأثر على التصميم |
|--------|---------------------|
| هل النظام لجيم واحد أم عدة فروع؟ | يحدد Multi-tenancy في نموذج البيانات |

| ما العملة والسوق المستهدف؟ | بوابة الدفع والامتثال الضريبي |
| هل مطلوب دعم اللغة العربية فقط أم ثنائي اللغة؟ | i18n من البداية أم لاحقًا |
